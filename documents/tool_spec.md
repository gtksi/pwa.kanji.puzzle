# KanjiVG・ミチムラ式 ハイブリッドデータ アノテーションツール 仕様書

## 1. ツール概要
オープンソースのKanjiVGデータとミチムラ式の部品リストを入力とし、Gemini API (3.1 Pro) に推論させる。そのマッピング結果および生成された字源・語彙をブラウザ上で視覚的に検証・修正し、アプリ本番用のJSONデータとして出力する社内用BFF (Backends For Frontends) アプリケーション。

## 2. 技術スタック
* Frontend: React, TypeScript, Tailwind CSS
* SVG Manipulation: ピュアなSVG DOM操作 (PixiJSは不使用。各 `<path>` 要素へのイベントバインドにより実現)
* Backend (BFF): Node.js (Express) または Next.js API Routes (Gemini APIキーの秘匿とCORS回避)
* LLM: Gemini 3.1 Pro API (Structured Output機能を活用した確定的なJSON生成)

## 3. ワークフローとUI構成 (3ペイン画面)
### 3.1. 左ペイン：入力・API制御
* 対象漢字の入力フィールド。
* ミチムラ式部品名の入力リスト（例：「方,𠂉,子,辶」のようにカンマ区切りで入力）。
* Gemini API推論実行トリガーボタン。
* 検証完了後のハイブリッドJSONエクスポートボタン。

### 3.2. 中央ペイン：形態レイヤー検証（Human-in-the-loop）
* 部品パレット: 現在選択中の部品をハイライト表示。
* KanjiVGキャンバス: 巨大な漢字を描画。キャンバス上のパス（画）をクリックすると、アクティブな部品グループにそのパスがアサインされ、色が即座に変わる。
* エラー防止: どの部品にも割り当てられていないパス（未割り当てパス）は、目立つ警告色（マゼンタ等）で表示し、マッピング漏れを防ぐ。

### 3.3. 右ペイン：意味レイヤー検証（テキストエディタ）
* 字源解説エディタ: Geminiが『説文解字』ベースで生成した字源のテキストエリア（教育的配慮による残酷表現のフィルタリングが適切か目視確認・修正する）。
* 語彙ネットワークエディタ: 生成された中学入試レベルの熟語（3件程度）と、その意味の編集フォーム。

## 4. 状態管理ロジック (State Schema)
Reactの宣言的UIモデルに基づき、中央ペインの描画を以下のステートで管理する。
* `activePartId` (型: string | null): 現在パレットで選択されている「塗り筆」の役割を果たす部品のID。
* `pathAssignments` (型: Record<string, string | null>): SVGパスのIDをキーとし、割り当てられた部品のIDをバリューとする辞書データ。ここを単一の信頼できる情報源（Single Source of Truth）としてSVG全体を再描画する。

## 5. システムプロンプト仕様 (バックエンドLLM推論)
ユーザーからの入力（漢字、学年、部品リスト）とKanjiVGのXMLデータを受け取り、以下のタスクを実行する。
* タスク1 (形態マッピング): KanjiVGのXMLパスを、指定されたミチムラ式部品へ論理的にマッピングする。
* タスク2 (字源生成): 11〜12歳向けに配慮した『説文解字』ベースの字源を生成する。
* タスク3 (語彙生成): 思考力を問う高度な文脈語彙ネットワークを生成する。
* 制約: 事前に定義したPydanticのJSONスキーマ（HybridKanjiData）に厳密に従って出力する。
